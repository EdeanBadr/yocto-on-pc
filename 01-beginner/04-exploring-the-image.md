# Exploring the Image

🟢 **Beginner**

You have a running embedded Linux system inside QEMU. This module walks you through what is actually inside it — and explains why it looks very different from a typical desktop Linux installation.

Boot into the image before starting (`runqemu qemux86-64 core-image-minimal nographic`, log in as `root`).

---

## The filesystem layout

```bash
ls /
```

```
bin  dev  etc  home  lib  lib64  media  mnt  proc  run  sbin  srv  sys  tmp  usr  var
```

This follows the [Filesystem Hierarchy Standard](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/index.html) — the same top-level structure as any Linux system. What differs is what is *inside* each directory.

---

## BusyBox — one binary for everything

```bash
which ls
# /bin/ls

ls -la /bin/ls
# lrwxrwxrwx 1 root root 7 ... /bin/ls -> busybox
```

Almost every command in `/bin` and `/usr/bin` is a **symlink to BusyBox** ([glossary](../docs/glossary.md#busybox)). BusyBox is a single binary that contains minimal implementations of ~300 Unix utilities. It checks `argv[0]` (the name it was called with) to decide which utility to emulate.

```bash
ls -la /bin | grep busybox | head -5
# lrwxrwxrwx 1 root root 7 ... cat -> busybox
# lrwxrwxrwx 1 root root 7 ... chmod -> busybox
# lrwxrwxrwx 1 root root 7 ... cp -> busybox
# ...

# All those symlinks resolve to one file:
ls -lh /bin/busybox
# -rwxr-xr-x 1 root root 560K ... /bin/busybox
```

560 KB delivers a functioning shell and most of what you need for a headless embedded system. No package manager, no man pages, no extras.

## Checking installed packages

Poky's `core-image-minimal` uses `rpm`, `deb`, or `ipk` packages depending on configuration. The default package format is `rpm`. Check what is installed:

```bash
rpm -qa
# You will see a short list — typically under 50 packages
```

Compare that to a desktop Ubuntu which has 1 500–2 000 packages installed by default. This image has only what was explicitly included in the image recipe.

## Processes

```bash
ps aux
```

```
PID   USER     COMMAND
    1 root     /sbin/init
    2 root     [kthreadd]
  ...
  105 root     /sbin/syslogd -n -O /var/log/messages
  108 root     /sbin/klogd -n
  112 root     -sh
  113 root     ps aux
```

Only a handful of processes run. The init system (PID 1) is either **SysVinit** or **systemd** depending on Poky's defaults for the Scarthgap release. `core-image-minimal` uses SysVinit by default to keep the image small.

### Checking the init system

```bash
file /sbin/init
```

If it prints something like `symbolic link to ../lib/systemd/systemd`, then systemd is the init. If it references `busybox` or `sysvinit`, it is SysVinit.

## Memory

```bash
free -m
```

```
              total        used        free      shared  buff/cache   available
Mem:            247          28         200           0          18         218
Total:          247          28         200
```

QEMU allocates 256 MB of RAM to the guest by default. The running system uses under 30 MB — leaving most RAM available for applications. This is the typical working envelope for a constrained embedded device.

## Storage

```bash
df -h
```

```
Filesystem      Size  Used Avail Use% Mounted on
/dev/root        30M   28M  1.3M  96% /
devtmpfs        124M     0  124M   0% /dev
tmpfs           124M     0  124M   0% /dev/shm
tmpfs           124M  156K  124M   1% /run
tmpfs           124M     0  124M   0% /tmp
```

The root filesystem is a 30 MB ext4 image (`core-image-minimal-qemux86-64.rootfs.ext4`). `devtmpfs` and `tmpfs` are memory-backed virtual filesystems — they do not consume disk space on the host.

Any files you write inside the guest are lost when QEMU exits unless you mount a persistent volume. QEMU's virtual disk is a copy-on-write overlay of the image file.

## The kernel

```bash
uname -r
# 6.6.x-yocto-standard

# Kernel was built by BitBake from the linux-yocto recipe:
cat /proc/version
```

The kernel is compiled from source by Yocto's `linux-yocto` recipe. The version and configuration are defined in the BSP layer (`meta-yocto-bsp`) for `qemux86-64`. You will learn to modify the kernel configuration in [`03-advanced/03-kernel-configuration.md`](../03-advanced/03-kernel-configuration.md).

## Where did these files come from?

Every file in the image was put there by a **recipe task**. The flow:

```
recipe (.bb)
  └─ do_install  →  ${D}/          (staging area per package)
       └─ do_package              (splits into sub-packages)
            └─ do_rootfs          (assembles rootfs from selected packages)
                 └─ do_image      (creates the .ext4 / .wic file)
```

To find which recipe installed a specific file:

```bash
# On the HOST (not inside QEMU), inside your build directory:
oe-pkgdata-util find-path /usr/bin/env
# coreutils: /usr/bin/env
```

`oe-pkgdata-util` is available after sourcing `oe-init-build-env`. It queries the package database written by BitBake during the build.

## Modifying the image

You cannot `apt-get install` or `pip install` inside this image — there is no package manager connected to a repository. To add software you have two options:

1. **Add a package to the image recipe** — rebuild the image with BitBake. This is the Yocto way and produces a reproducible result.
2. **Use `devtool` to develop and test iteratively** — covered in [`02-intermediate/05-devtool-workflow.md`](../02-intermediate/05-devtool-workflow.md).
