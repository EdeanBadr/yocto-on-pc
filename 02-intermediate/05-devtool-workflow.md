# The devtool Workflow

🟡 **Intermediate**

`devtool` is Yocto's developer tool for modifying recipes and their source code iteratively. Instead of editing a recipe, rebuilding the world, and guessing what changed, `devtool` extracts the source, lets you edit it, and feeds changes back — all while integrating with your normal BitBake build.

---

## When to use devtool

- Modifying an existing upstream recipe (e.g., patching a library from OE-Core)
- Developing a new recipe for your own source code with rapid iteration
- Upgrading a recipe to a newer upstream version
- Creating a patch file from local edits

`devtool` is available after sourcing `oe-init-build-env` — it lives in `poky/scripts/`.

---

## The workspace layer

When you run `devtool modify` or `devtool add`, devtool creates a special **workspace layer** at `build/workspace/`. This layer has a high priority, so its version of a recipe overrides everything else during development.

```bash
ls build/workspace/
# appends/   recipes/   sources/
```

The workspace is temporary. When you are done iterating, you use `devtool finish` to write the changes back into your permanent layer and remove the workspace entry.

---

## Workflow 1: Modifying an existing recipe

### Scenario

You want to add a configuration flag to `curl`.

### Step 1 — Extract the source

```bash
devtool modify curl
```

This:
1. Fetches and unpacks the `curl` source into `build/workspace/sources/curl/`
2. Applies all existing patches
3. Creates a git repository in that directory (so your edits are tracked as commits)
4. Adds a `.bbappend` to the workspace that points at the local source

```
INFO: Using source tree as build directory since no Makefile found
INFO: Recipe curl now set up to build from build/workspace/sources/curl/
```

### Step 2 — Make your changes

```bash
cd build/workspace/sources/curl/
# Edit source, configure scripts, whatever you need
vim src/config.h
```

Or apply a change via the configure flags. Edit the workspace `.bbappend`:

```bash
cat build/workspace/appends/curl_%.bbappend
```

```bitbake
# Automatically generated by devtool
FILESEXTRAPATHS:prepend := "${THISDIR}/${PN}:"
inherit externalsrc
EXTERNALSRC = "/home/you/yocto/build/workspace/sources/curl"
EXTERNALSRC_BUILD = "/home/you/yocto/build/workspace/sources/curl"
```

Add extra configure options:

```bash
echo 'EXTRA_OECONF:append = " --disable-ldap"' >> build/workspace/appends/curl_%.bbappend
```

### Step 3 — Build and test

```bash
bitbake curl
# Builds from your local workspace source
```

Boot your image and test. Iterate: edit → `bitbake curl` → test.

### Step 4 — Commit your changes as a patch

Once you are satisfied, commit your source edits in the workspace git repo:

```bash
cd build/workspace/sources/curl/
git add -p           # stage your changes
git commit -m "disable LDAP support for reduced attack surface"
```

### Step 5 — Write back to your layer

```bash
cd ~/yocto/build
devtool finish curl ~/yocto/meta-mylayer
```

`devtool finish`:
1. Extracts each new git commit as a `.patch` file
2. Copies the patches into your layer's recipe directory
3. Adds the patches to `SRC_URI` in a `.bbappend`
4. Removes the workspace entry for `curl`

Your layer now contains a clean `.bbappend` with the patch:

```
meta-mylayer/
└── recipes-support/
    └── curl/
        ├── curl_%.bbappend
        └── curl/
            └── 0001-disable-LDAP-support.patch
```

## Workflow 2: Adding a new recipe

### Scenario

You have a C project on your local disk at `~/myapp/` and want to create a Yocto recipe for it.

```bash
devtool add myapp ~/myapp/
```

`devtool add` inspects the source, detects the build system (`cmake`, `autotools`, `meson`, `make`, etc.), and generates a recipe skeleton in the workspace.

```
build/workspace/
├── recipes/
│   └── myapp/
│       └── myapp_git.bb    ← generated recipe
└── sources/
    └── myapp/              ← symlink to ~/myapp/
```

Edit the generated recipe to fill in `SUMMARY`, `LICENSE`, `LIC_FILES_CHKSUM`, and any build tweaks.

Build and iterate:

```bash
bitbake myapp
```

When satisfied:

```bash
devtool finish myapp ~/yocto/meta-mylayer
```

---

## Workflow 3: Upgrading a recipe

### Scenario

`curl` has a new upstream release and you want to update the recipe.

```bash
devtool upgrade curl --version 8.7.1
```

`devtool upgrade`:
1. Fetches the new version's source
2. Attempts to re-apply existing patches (reports conflicts if any)
3. Updates `PV` and `SRC_URI[sha256sum]` in the workspace recipe

```bash
# Review what changed
bitbake curl
# Fix any patch conflicts in build/workspace/sources/curl/
```

Write back:

```bash
devtool finish curl ~/yocto/meta-mylayer
```

## devtool deploy-target — testing on QEMU without a rebuild

Once you have built a package with `devtool`, you can push just that package's files directly into a running QEMU guest over SSH — without rebuilding the image:

```bash
# The QEMU guest must be running with SSH available (dropbear or openssh)
devtool deploy-target myapp root@192.168.7.2
```

This copies the installed files from the recipe's staging area directly to the target over `scp`. The change is live immediately, without rebooting or rebuilding the image.

When you are done testing:

```bash
devtool undeploy-target myapp root@192.168.7.2
```

## Useful devtool commands at a glance

```bash
devtool status                        # show all recipes in the workspace
devtool modify <recipe>               # extract source for editing
devtool add <name> <source-dir>       # create a new recipe from source
devtool upgrade <recipe> --version X  # bump a recipe to a new version
devtool build <recipe>                # build a workspace recipe
devtool deploy-target <recipe> user@host  # push files to a running target
devtool undeploy-target <recipe> user@host  # remove deployed files
devtool finish <recipe> <layer-path>  # write back to layer, clean workspace
devtool reset <recipe>                # abandon workspace changes, restore original
```
